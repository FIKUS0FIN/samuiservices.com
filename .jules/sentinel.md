## 2024-05-24 - [Fix IDOR in QA Server Actions]
**Vulnerability:** A critical IDOR (Insecure Direct Object Reference) vulnerability existed in `src/app/actions/qa.ts`. The `askQuestion` and `answerQuestion` server actions accepted `userId` as an argument directly from the client without verifying if it belonged to the currently authenticated user. This allowed any logged-in user to forge a request and post questions or answers impersonating any other user, including site administrators.
**Learning:** Next.js Server Actions can easily become vulnerable to IDOR if they blindly trust arguments passed from the client instead of extracting user identity from the server-side session. This is a common pitfall when migrating from traditional API routes where authentication middleware might have been used differently.
**Prevention:** Always extract the authenticated user's ID directly from `getServerSession(authOptions)` inside the server action itself. Do not rely on `userId` or similar identifiers passed from the client arguments or form data unless you are explicitly verifying that the authenticated user has permission to act on behalf of that ID.

## 2024-05-25 - [Prevent SSRF in AI Crawl & Image Upload]
**Vulnerability:** A Server-Side Request Forgery (SSRF) vulnerability existed in `src/app/api/ai/crawl/route.ts` and `src/app/api/upload-external-images/route.ts`. The endpoints accepted URLs from the client and fetched them on the server-side without verifying if the requested hostname was an internal IP or local domain. This could allow an attacker to scan internal networks, access cloud metadata services, or interact with internal services that the Next.js server has access to.
**Learning:** Next.js server functions (API routes or server actions) that fetch user-provided URLs must strictly validate the hostname to prevent SSRF. A regex check on the hostname is insufficient because it doesn't prevent DNS rebinding attacks or alternative IP encodings (like octal or decimal formats).
**Prevention:** To prevent SSRF effectively, especially against DNS rebinding, use a DNS lookup (`dns.lookup` in Node.js) to resolve the hostname to an IP address. Then, verify that the resolved IP address does not fall into private, loopback, or reserved IP ranges (e.g., `127.x.x.x`, `10.x.x.x`, `192.168.x.x`, `169.254.169.254`, `::1`) before allowing the connection. Checking for internal domains like `.local` and `.internal` is also recommended as a defense-in-depth measure.
